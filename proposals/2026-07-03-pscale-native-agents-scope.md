## The migration — genuinely slight, and exactly two seams

Everything above carries unchanged EXCEPT where the substrate dialect bites:

1. **Walker dialect.** `0` → `_` keys, plus the structural difference David names: the federated form has **hidden directories** (underscore-as-object, the `_._` fold, the `*` operator) which the biome deliberately lacks. Doctrine says never write another parser — so the kernel's walking routes through the EXISTING canonical `_`/1-9 walker via a thin adapter matching spark's call surface (`spark/voice/floor/parse/parse_reference`): `bsp2-star.py` for the Python kernel now; `src/bsp.ts` for the TS pulse when xstream needs it (P3). The spark engines stay untouched, 0-9 native, for the biome.
2. **Block I/O.** `load_block`/`save_block` on files → the beach wire (`?block=` GET / POST with secret), with the known write rules (identity floor, sectioned writes, digit-string positions) and retry-once / confirm-after-write (transport flake is real — see session note). `peers.json` → watched beaches; the between reads published `surface:<handle>` blocks (weft's and keel's are already live — the between is pre-populated). History gains `append=true` (server slot + supernest) — a free upgrade over slot-scan.

Pulse logic, heartbeat, digest/observer/resonance/trajectory, batteries: **carried, not rewritten**. The Python agent batteries keep guarding the kernel through the migration because the kernel stays Python; walker equivalence is guarded by the existing bsp batteries. A TS pulse is deferred to P3 (the browser animator needs it); it reuses `src/bsp.ts` and the same batteries.

## S\*T\*I and address-coupling — scoped correctly

S\*T\*I (same address across place/scene/character blocks) is a **biome** architecture — it emerged there precisely because the biome has no hidden directories. The federated `_`/1-9 form folds dimensions INTO the Spatial block via hidden directories — a different emergent architecture. Address-coupling therefore migrates as a **playable move, not a backbone**: v009 itself plays it (cadence/last-touched mirror purpose's addresses — that pattern carries), `bsp-floor` exists when the move is wanted (n-ary floor-aligned reads, e.g. `spatial:<world>` beside `spatial:<handle>` at one address), and nothing in the design REQUIRES it. Keep it in the vocabulary; reach for it when a design calls for perspectival or mirrored semantics; the spine rule (no metadata-on-node) holds regardless.
